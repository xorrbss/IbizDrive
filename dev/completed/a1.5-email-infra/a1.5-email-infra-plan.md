# a1.5-email-infra — Plan

Last Updated: 2026-05-02

## 요약

비밀번호 분실/재설정/변경 흐름과 그 전제가 되는 이메일 발송 인프라를 도입한다. 이메일 백엔드는 추상화하여 dev에서는 콘솔 logger, prod에서는 SMTP를 쓴다. 토큰은 SHA-256 해시 저장(평문은 메일에만), 30분 TTL, 1회용. 재설정 성공 시 해당 사용자의 모든 세션을 무효화한다.

## 현재 상태 분석

- `backend/src/main/resources/db/migration/`: V1~V7. 비밀번호 재설정 토큰 테이블 없음.
- `backend/build.gradle.kts`: spring-boot-starter-mail 미포함.
- `backend/src/main/java/com/ibizdrive/auth/`: 회원가입/로그인/로그아웃만 구현. 비밀번호 분실/재설정/변경 endpoint 없음.
- `frontend/src/app/`: `/login`, `/signup`만. forgot/reset 페이지 없음. UserMenu에 PW 변경 진입 없음.
- 의존하는 트랙: auth-pages(PR #43, AuthGuard/useMe/SignupService 제공). m-admin-entry는 본 트랙과 무관(직접 의존 없음).

## 목표 상태

1. Spring Mail + `EmailService` 추상화. 기본 빈 = `ConsoleEmailService`(stdout dump). `prod` 프로파일에서 `SmtpEmailService` 활성화.
2. V8 마이그레이션으로 `password_reset_tokens` 테이블 생성. token 평문은 저장하지 않고 SHA-256 해시(`token_hash`)만 저장.
3. `POST /api/auth/password/forgot` — 이메일 입력 → 가입자라면 토큰 생성+이메일 발송, 미가입자도 동일 200 응답(anti-enumeration).
4. `POST /api/auth/password/reset` — 토큰+새 PW. 성공 시 PW 갱신 + 모든 세션 무효화 + `used_at` 마킹.
5. `POST /api/auth/password/change` — 인증 사용자 + 현재 PW 일치 → PW 갱신. 현재 세션은 유지(다른 세션은 invalidate, P5 결정).
6. 프론트 `/forgot-password`, `/reset-password?token=...`, `/account/password` 페이지 + UserMenu 진입.
7. 문서: ADR(추상화/토큰 해시), docs/03 §2.7 + §4 audit, docs/02 §7 API.

## Acceptance Criteria

- (A) `forgot` 응답은 가입자/미가입자 모두 200 + 동일 메시지(타이밍/응답 차이 없음). 미가입자도 토큰을 생성하지 않고 즉시 200.
- (B) 모든 토큰은 `token_hash`로만 DB에 저장. 평문 토큰은 이메일 본문/링크에서만 노출.
- (C) `reset` 성공 시 같은 사용자의 모든 Spring Session 항목이 invalidate.
- (D) `change` 성공 시 현재 세션은 유지하되, 같은 사용자의 다른 세션은 invalidate.
- (E) `dev` 프로파일에서 SMTP 서버 없이도 forgot이 200을 반환하고 stdout에 메일 본문이 출력.
- (F) audit 이벤트 `password.forgot`(token issued), `password.reset`(success), `password.change`(success) 발생.
- (G) 토큰 TTL은 30분. 재사용 차단(used_at).

## 리스크 / 완화

- **이메일 미발송 누락**: dev에서는 정상 동작해 보이지만 prod에서 SMTP 미설정 시 무음 실패. → `SmtpEmailService`는 발송 실패를 audit + ERROR 로그로 명시. ConsoleEmailService는 prod 프로파일에서 활성화 금지.
- **anti-enumeration timing leak**: 미가입자 분기에서 `BCrypt.matches`나 토큰 생성을 건너뛰면 응답 시간이 짧아져 추측 가능. → 미가입자에서도 dummy work(BCrypt 1회) 수행 또는 응답 지연 일정화. 본 트랙은 일관 응답 시간 강제 대신 "미가입자도 토큰 INSERT 직전까지의 작업은 동일" 원칙으로 진행하고 ADR에 한계 명시.
- **세션 무효화 범위**: Spring Session JDBC `SPRING_SESSION` 테이블의 `principal_name`으로 사용자별 세션 일괄 삭제. → `JdbcIndexedSessionRepository.findByPrincipalName(email)`로 조회 + 각 세션 `delete`.
- **PW 정책 미정**: 본 트랙은 정책 강화 트랙과 분리. P4/P5는 현 정책(8자 이상 등 SignupService 기존 로직)을 재사용한다. 정책 강화 트랙이 따로 들어오면 `PasswordPolicyValidator` 클래스에서 단일 갱신.
