## 1. §2 인프라 게이트

### 1.1 전송 / 도메인 (§2.1)

- [ ] **HTTPS 종단** (리버스 프록시 또는 ALB) — TLS 1.2+ 강제
- [ ] **HSTS 헤더** — 리버스 프록시 또는 CDN 단에서 응답 추가 (`Strict-Transport-Security: max-age=31536000; includeSubDomains`)
- [ ] **사내 도메인 + DNS** 등록 (예: `ibizdrive.intra.<corp>.co.kr`)
- [ ] **CORS** — `ibizdrive.cors.allowed-origins`를 사내 도메인으로 override (현재 `application.yml` default `http://localhost:3000`)
  ```yaml
  # application-prod.yml override 또는 환경변수
  ibizdrive:
    cors:
      allowed-origins: "https://ibizdrive.intra.<corp>.co.kr"
  ```

### 1.2 쿠키 / 세션 (§2.2)

- [x] `application-prod.yml`에서 `server.servlet.session.cookie.secure=true` (mvp-prod-profile closure)
  → `SPRING_PROFILES_ACTIVE=prod` 환경변수만 설정하면 자동
- [x] `http-only=true`, `same-site=lax` (이미 설정 + `ProdProfileConfigTest` 회귀 차단)
- [ ] **확인**: prod 부팅 시 Set-Cookie 헤더에 `Secure; HttpOnly; SameSite=Lax` 모두 포함되는지 1회 점검

### 1.3 시크릿 (§2.3)

- [ ] **DB 패스워드 시크릿 주입** — `spring.datasource.password` 환경변수 또는 사내 시크릿 저장소
  - 현재 dev default(`dev_password_only_change_in_real_envs`) **운영 절대 금지**
  - 권장: `SPRING_DATASOURCE_PASSWORD` 환경변수 + sealed-secrets / Vault / AWS Secrets Manager
- [x] BCrypt strength=12 (`SecurityConfig.passwordEncoder()`, ADR #19)
- [ ] **`.env*` 파일 커밋 금지** — `.gitignore` 확인 (이미 차단되어 있으나 시크릿 추가 시 재확인)
- [ ] **Spring `SECRET_KEY` 등 추가 시크릿 부재 확인** — IbizDrive는 별도 JWT/HMAC 시크릿 미사용 (세션 쿠키 + DB-backed Spring Session)

### 1.4 DB / 스토리지 (§2.4)

#### Postgres (필수)
- [ ] **managed Postgres** (AWS RDS / GCP Cloud SQL / 사내 PG) + 자동 일일 스냅샷 + PITR (point-in-time recovery)
- [ ] **DB role 분리** (V4 마이그레이션 적용):
  - `app_user` — application 사용 (DML 권한, audit_log INSERT만)
  - `audit_admin` — audit_log SELECT 권한 + AUDITOR/ADMIN role 보유자에게 부여 (§8)
  - `db_superuser` — Flyway 마이그레이션 전용 (운영 후 권한 회수)
- [ ] **Flyway 마이그레이션 V1~V19 적용 확인**:
  ```bash
  # 부팅 시 자동. 수동 확인은:
  psql -c "SELECT version, description, success FROM flyway_schema_history ORDER BY installed_rank DESC LIMIT 5;"
  ```
- [ ] **V13 cutover 전제**: prod DB의 `folders` / `files` 테이블이 V13 적용 시점에 비어 있어야 함 (위 §0 참조)

#### Storage (필수)
- [ ] **storage 디스크 백업** — `ibizdrive.storage.local.root=./uploads` 경로 (또는 prod override 경로)
  - 권장: rsync + 일일 cron 또는 EBS snapshot
  - v1.x: S3 마이그레이션 트랙 (인프라 결정 선결 — S3 vs MinIO)

### 1.5 v1.x deferred (§2 게이트와 무관, 출시 후 트랙)

다음은 v1.x 후속 트랙으로 분리됨 (BETA-RELEASE.md §7 참조):

## 2. §3 운영 cron 활성화

- [x] `application-prod.yml`에서 cron 4종 모두 `enabled=true` (mvp-prod-profile closure)
  - `app.purge.enabled` — 휴지통 30일 후 hard delete
  - `app.share.expiration.enabled` — 만료 공유 자동 expire + audit
  - `app.permission.expiration.enabled` — 만료 권한 cleanup + audit
  - `app.storage.orphan-cleanup.enabled` — 일일 storage 객체 회수 (ADR #38)
- [ ] **prod 부팅 후 첫 24시간 cron dry-run 점검**:
  - `purge.expired`: 휴지통에서 30일 경과 row가 hard delete됐는지 (audit `FILE_PURGED` / `FOLDER_PURGED` emit 확인)
  - `share.expire`: 만료 공유 자동 만료 처리 + `SHARE_EXPIRED` emit
  - `permission.expire`: 만료 권한 cleanup + `PERMISSION_EXPIRED` emit
  - `storage.orphan.cleanup`: orphan 객체 회수 + `STORAGE_ORPHAN_CLEANED` emit
- [ ] **단일 인스턴스 가정** — 멀티 인스턴스 시 `@SchedulerLock` 도입 필요 (ADR backlog)

## 5. 롤백 절차

베타 출시 후 critical 이슈 발생 시:

1. **즉시 차단**: 리버스 프록시 단에서 503 / maintenance 페이지 노출 (5분 내)
2. **롤백 결정**: 운영팀 + 인프라팀 동의 → 직전 stable 태그로 재배포
3. **DB 롤백**: V19 → V18 등 schema 롤백은 **별도 결정 필요** (Flyway down migration 미제공). 대신 hot-patch 우선.
4. **audit_log 보존**: 롤백 후에도 audit_log는 append-only로 보존 (DB-level REVOKE UPDATE/DELETE 강제)

자세한 incident response 절차는 v1.x 운영 런북 트랙 (별도).
